This page covers the [operator integration](/docs/operator-integration/launch) path. Aggregator-path security (Bearer inbound, SHA-512 outbound signatures) is described on [Aggregator integration](/docs/aggregator-integration/overview).

## Requirements

| Requirement | Detail |
| --- | --- |
| TLS | HTTPS on every API call |
| Protocol version | TLS 1.2 or newer; TLS 1.3 preferred |
| Credential storage | Treat launch and callback Basic Auth passwords like production secrets; rotate if leaked |
| Session isolation | Do not expose `SessionId` to the game client |
| Idempotency | Key wallet handling on `ExternalTransactionId` |
| Launch usage | Request one launch URL per player session. Do not poll the endpoint |
| Logging | Never log launch passwords, callback passwords, launch JWTs, full iframe URLs, or raw wallet payloads containing player ids |

## Operational controls

- Launch and session validation must fail for frozen, self-excluded, jurisdiction-blocked, or high-risk accounts.
- Support tools must not expose launch URLs, credentials, or full wallet payloads.
- Logs must redact secrets and use `ExternalTransactionId` + `RoundId` for reconciliation.
- Wallet idempotency state must survive service restarts, deploys, and pod failover.

## Launch tokens

Launch URLs carry a JWT that expires in about 30 seconds. Request the URL right before embedding it in the iframe.

- Request a fresh URL per session; do not cache or share it.
- The short lifetime is the control today. The token is replayable within that window, so keep the URL server-to-client only and never log it.
- Balance is not in the JWT; we pull it from your wallet API during play.

## Idempotency and ledger safety

Store each `ExternalTransactionId` when you process it. If the same ID arrives again, return the stored `TransactionId` and `CurrentBalance` without re-processing. We never send the same `ExternalTransactionId` with a different `Amount`, `Type`, or `RoundId`.

Implement idempotency atomically in your database, not with an in-memory lock. We may call your wallet for the same player concurrently, and retries can land on a different pod. Concurrent duplicate bets must result in one debit and one stored response.

## IP allowlisting

If you want to lock down wallet callbacks to our egress IPs, ask during onboarding. We can send the addresses to allowlist. IP allowlisting is defense in depth; it does not replace Basic Auth.

## Secret rotation

Keep separate credentials for staging and production. Rotate immediately if a password, launch URL, or JWT appears in logs, tickets, analytics, or screenshots.

Preferred rotation pattern:

1. Add the new credentials while the old ones still work.
2. Switch traffic to the new credentials.
3. Confirm launch and wallet calls in staging/production logs.
4. Disable the old credentials.

## Player data and support tooling

Use opaque player ids. Do not send PII unless we explicitly agree it is required. For support and reconciliation, use `ExternalTransactionId` and `RoundId`; these are safer and more precise than screenshots with full payloads.

## Responsible gaming and blocking

If a player is blocked, self-excluded, cooling off, over a limit, or outside an allowed jurisdiction, return `ErrorCode: 111` from session validation or wallet calls. Do not let the player open a new Spark session and then fail only after the first bet.

## This documentation site

Run it behind an access gate (Cloudflare Zero Trust works well). Do not leave it public. Credentials always go out-of-band, never in these pages.
