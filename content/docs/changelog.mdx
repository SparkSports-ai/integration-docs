## 2026-07-01

Clarifications to the current live (r0) contract. None of these are API changes.

| Area | Change |
| --- | --- |
| Docs | Retry policy: we retry only on `ErrorCode: 99`, up to 3 total attempts with 1s and 2s delays. Network failures and 5xx without `ErrorCode: 99` are not retried. |
| Docs | `RoundId` is a UUID. One is generated per streak. |
| Docs | Number precision: `Amount`, `Balance`, and `CurrentBalance` are JSON numbers with two decimal places. Do not send integer cents or strings. |
| Docs | Concurrency: we may call your wallet for the same player concurrently. Handle per-player concurrency safely. |
| Docs | Launch errors: `OperatorAuthFailed` returns your authenticate endpoint's HTTP status. Added `OperatorUnreachable` and `OperatorAuthInvalid` (both 502). |
| Docs | Error handling: any 2xx is accepted as success. Unrecognized `ErrorCode` values are treated as a generic, non-retryable operator error. |
| Docs | Session validation: `Country` is optional. `Balance`, `Username`, and `Currency` are required. We do not retry authenticate. |

## v1.0 (draft): 2026-06-30

Proposed modernization of the integration contract. Consolidates our direct-operator and aggregator integrations into one clean API. **Not yet live**: items below need backend work before partners use them.

- **Auth:** Bearer token everywhere, plus signed wallet/auth callbacks (`X-Spark-Signature`, `X-Spark-Timestamp`) using HMAC-SHA256 over `{timestamp}.{METHOD}.{path-and-query}.{rawBody}`. Replaces HTTP Basic Auth.
- **Correlation:** every callback carries `X-Spark-Request-Id` for support and reconciliation.
- **Money:** all amounts are decimal strings; JSON numbers are not part of the v1 contract.
- **Currency:** carried on the authenticate response, balance response, and every transaction. Wallet calls must match the session currency.
- **Transaction types:** words `BET` / `WIN` / `ROLLBACK` instead of numeric `1` / `2` / `3`.
- **Error codes:** string codes (`INSUFFICIENT_BALANCE`, `SESSION_EXPIRED`, `CURRENCY_MISMATCH`, `ROUND_ALREADY_SETTLED`, `RESPONSIBLE_GAMING_BLOCK`, `RATE_LIMITED`, ...) instead of numeric (`104`, `102`, ...); HTTP status is now the single routing signal.
- **Linkage:** `WIN` / `ROLLBACK` reference the originating `BET` via `betTransactionId`. Losing round-close carries `reason` + `roundFinished`.
- **Naming:** one consistent `playerId` (was `Pincode` / `PinCode`); `transactionId` (was `ExternalTransactionId`); dropped the constant `GameId`.
- **Session profile:** optional `limits` let operators narrow stake bounds at launch; the stricter of configured limits and session limits applies.
- **Launch:** adds `expiresAt` (absolute UNIX timestamp) to the response; corrected the inaccurate "single-use" claim.
- **Idempotency, timeouts:** specified precisely (dedupe key, mismatch handling, durable dedupe retention, concurrent duplicate behavior, 10s wallet timeout).
- **Docs:** Mermaid diagrams for every flow.

## r0: current live contract

The r0 direct integration: HTTP Basic Auth, numeric `Type` (`1`/`2`/`3`), numeric `ErrorCode`, no currency on transactions, launch response `{ url }`. Still what live operators use until v1 ships. We will keep a compatibility appendix for them.
