---
title: Session validation
description: Endpoint your platform must expose so SparkSports can validate player sessions.
---

We call your backend before handing out a launch URL. If the session is not valid, the launch fails.

## Your endpoint

```
POST {your-callback-base-url}/api/generic/authenticate
```

You tell us the callback base URL during onboarding.

## Authentication

**Bearer token** plus the same request signature as the [wallet callbacks](/docs/direct-integration/wallet-api#authentication). Validate the signature before trusting `sessionId`.

## Request

```json
{
  "sessionId": "your-player-session-id"
}
```

## Success response

```json
{
  "sessionId": "your-player-session-id",
  "playerId": "player-unique-id",
  "username": "john_doe",
  "balance": "1500.00",
  "currency": "USD",
  "country": "US",
  "limits": {
    "minStake": "1.00",
    "maxStake": "100.00"
  }
}
```

| Field | Required | Description |
| --- | --- | --- |
| `sessionId` | Yes | Same session id we sent |
| `playerId` | Yes | Stable player id on your side. Used as the key on every later wallet call |
| `username` | Yes | Display name |
| `balance` | Yes | Wallet balance at launch, as a decimal string |
| `currency` | Yes | ISO 4217 code (`USD`, `EUR`, ...) |
| `country` | No | ISO 3166-1 alpha-2 code |
| `limits` | No | Operator-side stake limits if they are narrower than the configured Spark limits |

`playerId` must be stable and non-PII. Do not send email, phone, government ID, or a raw account number as `playerId`. If your internal id is sensitive, send an opaque stable alias.

The `currency` returned here is the wallet settlement currency for the session. Later wallet calls for the same session must use the same currency.

If `limits` are returned, Spark applies the stricter of the configured Spark limits and the session response limits. Session response limits may narrow configured limits, but they cannot widen them.

## Error response

```json
{
  "error": {
    "code": "SESSION_EXPIRED",
    "message": "Session expired"
  }
}
```

| `error.code` | Meaning |
| --- | --- |
| `SESSION_EXPIRED` | Session invalid or expired |
| `PLAYER_BLOCKED` | Player blocked |
| `RESPONSIBLE_GAMING_BLOCK` | Player is blocked by cool-off, self-exclusion, jurisdiction, or limit policy |
| `UNAUTHORIZED` | Bad bearer token or signature (usually a config mismatch) |

## Notes

- `playerId` is the player key for all later wallet calls.
- Balance at authenticate is a snapshot. During play we read balance from your [wallet API](/docs/direct-integration/wallet-api).
- Keep `sessionId` on your backend. Do not pass it into client-side game code.
- If a session expires while the iframe is open, return `SESSION_EXPIRED` from wallet calls; we will stop play and ask the player to relaunch.
