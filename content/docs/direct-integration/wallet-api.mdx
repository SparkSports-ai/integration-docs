---
title: Wallet API
description: Endpoints your platform must expose for bets, wins, and rollbacks.
---

While someone is playing Spark, we call your wallet API to move money. Every call is synchronous: we send the request, wait for your response, then continue or show an error to the player.

## Authentication

HTTP Basic Auth on all wallet endpoints, using the callback credentials from onboarding.

## Get balance

We call this when we need a fresh balance during play.

```
GET {your-callback-base-url}/api/generic/user/{pincode}/balance?sessionId={sessionId}
```

Response:

```json
{
  "Balance": 1500.00
}
```

## Process transaction

One endpoint for bets, wins, and rollbacks.

```
POST {your-callback-base-url}/api/transaction/process
```

Request body:

```json
{
  "Amount": 10.00,
  "Type": 1,
  "ExternalTransactionId": "spark-tx-abc123",
  "Description": "Spark streak bet",
  "GameId": "sparksports",
  "RoundId": "round-uuid-here",
  "PinCode": "player-unique-id",
  "SessionId": "your-player-session-id"
}
```

| Field | Description |
| --- | --- |
| `Amount` | Always positive |
| `Type` | `1` bet (debit), `2` win (credit), `3` rollback (refund) |
| `ExternalTransactionId` | Our transaction ID; use this for idempotency |
| `Description` | Free text |
| `GameId` | Always `"sparksports"` |
| `RoundId` | UUID for the round |
| `PinCode` | Player id from authenticate (capital `C` here; authenticate returns it as `Pincode`) |
| `SessionId` | Active player session |

Response:

```json
{
  "TransactionId": 987654321,
  "CurrentBalance": 1490.00,
  "ExternalTransactionId": "spark-tx-abc123"
}
```

## Idempotency

Same `ExternalTransactionId` twice must return the same result without processing twice. See [Retries](#retries).

## Errors

To signal a failure, return a non-2xx HTTP status with this body. We route on `ErrorCode`, not the HTTP status.

```json
{
  "ErrorCode": 104,
  "Message": "Insufficient balance"
}
```

| ErrorCode | Meaning |
| --- | --- |
| `104` | Insufficient balance |
| `125` | Invalid amount |
| `111` | Player blocked |
| `102` | Session expired |
| `122` | Invalid pincode or session |
| `126` | Bad callback credentials |
| `99` | Operator internal error (we may retry this) |

## Retries

We retry transient operator errors (`ErrorCode 99`) up to three times with backoff (1s, 2s, 4s). Settled business outcomes like insufficient balance, blocked player, or expired session are never retried; they go straight to the player. Make your handler idempotent on `ExternalTransactionId` so any retry returns the same result.
