While someone is playing Spark, we call your wallet API to move money. Every call is synchronous: we send the request, wait for your response, then continue or show an error to the player.

<Callout type="info">**Proposed for v1.** The auth, currency, string types, and linkage fields on this page are the target v1 contract and need backend work before they go live. See the [Changelog](/docs/changelog).</Callout>

## Authentication

{/* v1: was "HTTP Basic Auth". WHY change: Basic Auth ships a static password in every request and only authenticates the channel. A Bearer token plus a per-request signature authenticates the message itself, so a callback cannot be replayed or tampered. Our Upgaming integration uses an SHA-512 signature; v1 deliberately standardizes on HMAC-SHA256 (the modern default; for HMAC it is not weaker than SHA-512). Pick one and keep it consistent across the contract. */}

**Bearer token** on every wallet call (`Authorization: Bearer <token>`), plus a **request signature**. We sign the raw request target, timestamp, and raw request body with a shared secret. Signatures provide integrity and authenticity, not confidentiality; payload confidentiality comes from TLS. There is no payload encryption and nothing for you to decrypt.

Request headers:

{/* v1: NEW header table. WHY: betting operators need an exact checklist for gateway/middleware implementation, and support needs one stable request id to join Spark logs with operator logs. */}

| Header | Description |
| --- | --- |
| `Authorization` | `Bearer <token>` |
| `X-Spark-Timestamp` | UNIX timestamp in seconds |
| `X-Spark-Signature` | Hex HMAC-SHA256 signature |
| `X-Spark-Request-Id` | Correlation id for support and reconciliation |

Canonical string:

{/* v1: NEW canonical string. WHY: HMAC schemes fail interoperability when each side guesses a different canonicalization; binding method + path prevents replaying a valid signed body to a different endpoint. */}

```text
{timestamp}.{METHOD}.{path-and-query}.{rawBody}
```

Verify all of it:

{/* v1: NEW verification checklist. WHY: timestamp bounds replay, transactionId idempotency prevents duplicate money movement, and raw-body verification prevents JSON parser differences from changing what was signed. */}

- Recompute the HMAC over the exact raw body bytes you received; do not re-serialize JSON before verifying.
- Reject if the signature does not match.
- Reject if `X-Spark-Timestamp` is more than 60 seconds away from your server clock.
- Log `X-Spark-Request-Id`, `transactionId`, `roundId`, and `playerId` on every wallet call.

{/* v1: intentionally dropped X-Spark-Nonce. WHY: request replay is bounded by timestamp, wallet mutations are protected by transactionId idempotency, and balance calls are read-only. Removing nonce avoids requiring every partner to operate a distributed nonce store across pods; X-Spark-Request-Id stays correlation-only and may be reused across retries. */}

### Signing a request

Spark signs callback requests. You verify them by rebuilding the same canonical string and comparing the signature.

1. `timestamp` is the current UNIX time in whole seconds, as a string.
2. `path-and-query` is the request path including the leading `/` and the full query string exactly as sent. Do not include the host or a URL fragment. If a proxy may rewrite the URL, sign the original path and query.
3. `rawBody` is the exact UTF-8 bytes of the request body. For `GET` balance calls with no body, use an empty string.
4. Build the canonical string by joining these values with literal `.` dots in this order: `timestamp`, `METHOD` uppercase, `path-and-query`, `rawBody`.
5. `X-Spark-Signature` is the lowercase hex output of `HMAC-SHA256(secret, canonicalString)`, where `secret` is the shared signing secret as UTF-8 bytes.

Do not re-serialize JSON before signing or verifying. Sign and verify the raw bytes exactly as they appear on the wire.

{/* v1: clarified signing ownership. WHY: Spark signs wallet callbacks and partners verify them; partners do not sign these callbacks back to us. The byte-level procedure still matters because the verifier must rebuild the exact same canonical string. */}

### Verifying a request (partner side)

Read the raw body before any JSON middleware parses or re-serializes it. Return `401` on any verification failure. Only parse the body as trusted after the bearer token, signature, and timestamp checks pass.

Node.js:

```js
import { createHmac, timingSafeEqual } from 'node:crypto';

function hmacHex(secret, canonical) {
  return createHmac('sha256', Buffer.from(secret, 'utf8'))
    .update(Buffer.from(canonical, 'utf8'))
    .digest('hex');
}

function signaturesMatch(expectedHex, receivedHex = '') {
  const expected = Buffer.from(expectedHex, 'hex');
  const received = Buffer.from(receivedHex, 'hex');
  return expected.length === received.length && timingSafeEqual(expected, received);
}

function verifySparkRequest({ method, pathAndQuery, headers, rawBody, secret, nowSeconds = Math.floor(Date.now() / 1000) }) {
  const timestamp = headers['x-spark-timestamp'];
  const receivedSignature = headers['x-spark-signature'];

  if (!timestamp || !receivedSignature) return { ok: false, status: 401 };

  const timestampNumber = Number(timestamp);
  if (!Number.isInteger(timestampNumber) || Math.abs(nowSeconds - timestampNumber) > 60) {
    return { ok: false, status: 401 };
  }

  const canonical = [timestamp, method.toUpperCase(), pathAndQuery, rawBody].join('.');
  const expectedSignature = hmacHex(secret, canonical);
  if (!signaturesMatch(expectedSignature, receivedSignature)) return { ok: false, status: 401 };

  return { ok: true, status: 200 };
}

const rawBody =
  '{"type":"BET","amount":"10.00","currency":"USD","transactionId":"spark-tx-abc123","roundId":"round-uuid-here","playerId":"player-unique-id","sessionId":"your-player-session-id"}';

const result = verifySparkRequest({
  method: 'POST',
  pathAndQuery: '/api/transaction/process',
  rawBody,
  secret: 'whsec_spark_example_0123456789abcdef',
  nowSeconds: 1782763200, // test-vector only; production uses the default current time
  headers: {
    'x-spark-timestamp': '1782763200',
    'x-spark-signature': '85cca7b4b84380c525b48a4a8a7f8f3b20e0f68f23aeaeffbe92758928d93dc3',
  },
});

console.log(result.ok ? 'valid' : `reject ${result.status}`);
// valid

// if (!result.ok) return new Response('Unauthorized', { status: 401 });
// const body = JSON.parse(rawBody);
```

Python:

```python
import hmac
import hashlib
import time

def hmac_hex(secret, canonical):
    return hmac.new(
        secret.encode("utf-8"),
        canonical.encode("utf-8"),
        hashlib.sha256,
    ).hexdigest()

def verify_spark_request(method, path_and_query, headers, raw_body, secret, now_seconds=None):
    now_seconds = int(time.time()) if now_seconds is None else now_seconds
    timestamp = headers.get("x-spark-timestamp")
    received_signature = headers.get("x-spark-signature")

    if not timestamp or not received_signature:
        return False, 401

    try:
        timestamp_number = int(timestamp)
    except ValueError:
        return False, 401

    if abs(now_seconds - timestamp_number) > 60:
        return False, 401

    canonical = ".".join([timestamp, method.upper(), path_and_query, raw_body])
    expected_signature = hmac_hex(secret, canonical)
    if not hmac.compare_digest(expected_signature, received_signature):
        return False, 401

    return True, 200

raw_body = '{"type":"BET","amount":"10.00","currency":"USD","transactionId":"spark-tx-abc123","roundId":"round-uuid-here","playerId":"player-unique-id","sessionId":"your-player-session-id"}'

ok, status = verify_spark_request(
    method="POST",
    path_and_query="/api/transaction/process",
    raw_body=raw_body,
    secret="whsec_spark_example_0123456789abcdef",
    now_seconds=1782763200,  # test-vector only; production uses the default current time
    headers={
        "x-spark-timestamp": "1782763200",
        "x-spark-signature": "85cca7b4b84380c525b48a4a8a7f8f3b20e0f68f23aeaeffbe92758928d93dc3",
    },
)

print("valid" if ok else f"reject {status}")
# valid
```

## Timeouts

{/* v1: NEW. WHY: the calls are synchronous and we block on them, but the old doc never stated how long we wait. Operators need a number to design against. */}

We wait **up to 10 seconds** for a wallet response. No response in time is treated as a transient failure and retried (see [Retries](#retries)). Keep wallet handlers fast; a healthy endpoint should normally respond in under 1 second.

## Get balance

We call this when we need a fresh balance during play.

```
GET {your-callback-base-url}/api/generic/user/{playerId}/balance?sessionId={sessionId}
```

Response:

```json
{
  "balance": "1500.00",
  "currency": "USD"
}
```

{/* v1: added "currency" to the balance response (was just { Balance }). WHY: a bare number is ambiguous for multi-currency operators; we should never read a balance without knowing its currency. */}

Amounts are decimal strings. Do not serialize money amounts as JSON numbers, and do not use binary floating-point internally for wallet math.

{/* v1: decimal strings are mandatory for money. WHY: allowing JSON numbers reopens the precision bug this spec is trying to prevent; betting wallet integrations fail badly on precision drift, especially when crypto currencies or non-2dp fiat-like units appear. */}

## Process transaction

One endpoint for bets, wins, and rollbacks.

```
POST {your-callback-base-url}/api/transaction/process
```

Request body:

```json
{
  "type": "BET",
  "amount": "10.00",
  "currency": "USD",
  "transactionId": "spark-tx-abc123",
  "roundId": "round-uuid-here",
  "playerId": "player-unique-id",
  "sessionId": "your-player-session-id",
  "description": "Spark streak bet"
}
```

{/* v1: type is now a word "BET" (was numeric 1). amount now carries "currency". field is "transactionId" (was ExternalTransactionId) and "playerId" (was PinCode/Pincode, which the old doc itself flagged as an inconsistent-casing bug). All renamed for one consistent casing. */}

| Field | Type | Description |
| --- | --- | --- |
| `type` | string | `BET` (debit), `WIN` (credit), `ROLLBACK` (refund) |
| `amount` | decimal string | Always positive. `0` is valid on a `WIN` that closes a losing round |
| `currency` | string | ISO 4217 of the amount |
| `transactionId` | string | Our transaction id. Idempotency key |
| `roundId` | string | UUID for the round |
| `playerId` | string | Player id from authenticate |
| `sessionId` | string | Active player session |
| `betTransactionId` | string | On `WIN`/`ROLLBACK` only: the `transactionId` of the `BET` this settles or reverses |
| `reason` | string | On `WIN` only: `cashout`, `loss`, or `cap` |
| `roundFinished` | boolean | On `WIN` only: `true` when this win closes the round |
| `description` | string | Free text |

### Settlement rules

{/* v1: NEW settlement rules. WHY: these are the invariants that prevent double settlement, currency drift, and ambiguous zero-amount wins during reconciliation. */}

- `BET` creates exposure and debits `amount`.
- `WIN` must reference the original `BET` with `betTransactionId`.
- `WIN` with `reason: loss` must have `amount: "0"` and `roundFinished: true`.
- `WIN` with `reason: cashout` or `reason: cap` credits `amount` and closes the round when `roundFinished: true`.
- `ROLLBACK` must reference the original `BET` with `betTransactionId` and refund exactly that bet amount.
- `currency` must match the player session currency and the original `BET`.
- Reject duplicate settlement: after a round has a closing `WIN` or `ROLLBACK`, another settlement for the same `BET` must not move money.

{/* v1: NEW fields betTransactionId, reason, roundFinished. WHY: (1) a WIN/ROLLBACK never referenced the originating BET before, so settlement/audit could not link them. (2) reason + roundFinished make the amount=0 loss-close self-documenting instead of a bare zero. This mirrors Upgaming's isroundfinished + reason, which their spec requires. GameId was dropped: it was a constant "sparksports" shipped in every call, pure noise. */}

Response:

```json
{
  "operatorTransactionId": "987654321",
  "balance": "1490.00",
  "currency": "USD",
  "transactionId": "spark-tx-abc123"
}
```

{/* v1: response renamed TransactionId -> operatorTransactionId (it is the operator-side id, and we persist it for reconciliation), CurrentBalance -> balance, added currency, echoes our transactionId. */}

## Idempotency

{/* v1: was one vague line. WHY: an idempotency contract has to define the window, the dedupe key, and the amount-mismatch case, or operators implement it three different ways. */}

Dedupe on `transactionId`. A repeat of a `transactionId` you have already applied must return the **same response** as the first, without moving money again. If the same `transactionId` arrives with any different immutable field (`type`, `amount`, `currency`, `roundId`, `playerId`, `sessionId`, `betTransactionId`), reject it with `DUPLICATE_MISMATCH` rather than guessing.

Keep the dedupe record for at least 30 days. Operators often need longer retention for audit and dispute handling; 90 days is preferred if your platform supports it.

{/* v1: changed minimum retention from 24h to 30d. WHY: payment disputes, delayed retries, and reconciliation investigations commonly outlive one day; a one-day dedupe window is too short for a betting ledger. */}

Implement idempotency atomically in your database, not with an in-memory lock. Concurrent duplicate `BET` requests must result in one debit and one stored response.

{/* v1: NEW atomicity note. WHY: wallet APIs are usually horizontally scaled; process-local locks do not protect against retries landing on another pod. */}

## Balance and exposure

Balance responses should include only settled wallet balance. Do not subtract a pending bet twice if the same `BET` is retried. If your wallet has reserved/bonus balances, agree the exposure model during onboarding and keep cash and bonus balances distinguishable in your internal ledger.

{/* v1: NEW balance/exposure section. WHY: many casino wallets split cash, bonus, reserved, and playable balances; without an agreed model, Spark can show or validate the wrong funds. */}

## Errors

{/* v1: resolved the old contradiction ("return a non-2xx status" AND "we route on ErrorCode, not the HTTP status"). WHY: two sources of truth is ambiguous. Now the HTTP status is authoritative for routing, and error.code is a self-documenting string for the specific reason. Numeric codes (104, 125, ...) become strings. */}

The **HTTP status is the signal**:

- `200` — applied
- `4xx` — business rejection, we will **not** retry, the player sees it
- `5xx` or timeout — transient, we retry

Include a body so we can show the right message:

```json
{
  "error": {
    "code": "INSUFFICIENT_BALANCE",
    "message": "Insufficient balance"
  }
}
```

| `error.code` | HTTP | Meaning |
| --- | --- | --- |
| `INSUFFICIENT_BALANCE` | 400 | Not enough funds for the bet |
| `INVALID_AMOUNT` | 400 | Amount is negative, zero where not allowed, or wrong precision |
| `CURRENCY_MISMATCH` | 400 | Request currency does not match the player session or original bet |
| `DUPLICATE_MISMATCH` | 409 | Reused `transactionId` with a different amount or type |
| `ROUND_ALREADY_SETTLED` | 409 | A `WIN` or `ROLLBACK` already closed this round |
| `PLAYER_BLOCKED` | 403 | Player cannot transact |
| `SESSION_EXPIRED` | 401 | Session no longer valid |
| `INVALID_PLAYER_OR_SESSION` | 401 | Unknown `playerId` or `sessionId` |
| `UNAUTHORIZED` | 401 | Bad bearer token or signature |
| `OPERATOR_ERROR` | 500 | Internal error on your side (we may retry) |

{/* v1: mapping table is the BetLive numeric codes translated to strings, plus DUPLICATE_MISMATCH and the signature failure. Keep the legacy numeric codes in a compatibility appendix for operators already live on them. */}

## Retries

We retry transient failures (HTTP `5xx`, timeout) up to **three times** with backoff (1s, 2s, 4s). Settled business outcomes (`4xx`: insufficient balance, blocked player, expired session) are never retried; they go straight to the player. Make your handler idempotent on `transactionId` so any retry returns the same result.

If you timed out but later applied the transaction, the retry must return the applied response. Never answer a timed-out duplicate with "unknown" after money moved.

{/* v1: NEW timeout-after-apply rule. WHY: the worst wallet failure mode is "money moved but retry says unknown", which creates reconciliation drift and bad player messaging. */}
