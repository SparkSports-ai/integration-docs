We call your backend before handing out a launch URL. If the session is not valid, the launch fails.

<Callout type="info">**Proposed: v1 (draft).** A future contract replaces Basic Auth with a Bearer token plus a request signature, and numeric `ErrorCode` with string error codes. Not live yet. See the [Changelog](/docs/changelog).</Callout>

## Your endpoint

```
POST {your-callback-base-url}/api/generic/authenticate
```

You tell us the callback base URL during onboarding.

## Authentication

HTTP Basic Auth with the callback credentials we configure together. We send them on every call.

## Request

```json
{
  "SessionId": "your-player-session-id"
}
```

Replay this against your handler to test your implementation:

```bash
curl -X POST "$CALLBACK_BASE/api/generic/authenticate" \
  -u "$CALLBACK_USER:$CALLBACK_PASS" \
  -H "Content-Type: application/json" \
  -d '{"SessionId":"your-player-session-id"}'
```

## Error response

```json
{
  "ErrorCode": 102,
  "Message": "Session expired"
}
```

| ErrorCode | Meaning |
| --- | --- |
| `102` | Session invalid or expired |
| `111` | Player blocked |
| `126` | Bad callback credentials (usually a config mismatch on our side) |

We do not retry authenticate. One attempt; if it fails, launch fails.

## Notes

- `Pincode` is the player key for all later wallet calls.
- `Balance`, `Username`, and `Currency` are required. If any is missing, launch fails with `OperatorAuthInvalid`.
- `Country` is optional. If you omit it, we store an empty value.
- Balance at authenticate is a snapshot. During play we read balance from your [wallet API](/docs/operator-integration/wallet-api).
- Keep `SessionId` on your backend. Do not pass it into client-side game code.
