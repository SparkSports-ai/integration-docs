This is a **proposed v1 of the Spark integration contract**, consolidating what we
learned running both a direct-operator (Basic Auth, numeric codes) and an aggregator
(Bearer, signed callbacks) integration into one clean API.

**What changed and why:**
- **Bearer auth + signed callbacks** instead of Basic Auth. Callback signatures bind
  timestamp, method, path, and raw body with HMAC-SHA256.
- **Decimal-string money + currency** on every money object. Wallet calls must match the
  session currency.
- **Words instead of numbers** for transaction types (`BET` / `WIN` / `ROLLBACK`) and
  for error codes (`INSUFFICIENT_BALANCE`, `SESSION_EXPIRED`, ...), so the wire is
  self-documenting.
- **Explicit linkage**: a `WIN` / `ROLLBACK` references the originating `BET`, and a
  round-close carries a `reason` (`cashout` / `loss` / `cap`). A closed round cannot be
  settled again.
- **Durable idempotency, stated timeouts, and timeout-after-apply retry behavior** for
  wallet calls.

Spark is SparkSports' live sports streak game. Players pick outcomes across consecutive live markets, build a multiplier, and cash out before the streak ends.

If you operate a casino platform, you need three pieces in place:

1. A way to request a game URL when a player opens Spark
2. An endpoint that confirms the player session is still valid
3. Wallet endpoints that handle bets, wins, and rollbacks

There is no SDK and no webhooks. Everything runs over HTTPS with plain request/response calls.

## Player experience

Spark loads in an iframe in your lobby. The browser only gets a short-lived launch URL. Wallet credentials stay on your backend.
